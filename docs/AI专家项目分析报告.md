## 一、项目定位与核心价值

### 1.1 项目本质

这**不是**一个简单的 AI 聊天机器人，而是一个**面向汽车 FOTA（固件空中升级）故障诊断的多智能体根因分析系统**。

**核心护城河**：
- ✅ 多源异构日志的结构化解析能力（Android/Kernel/MCU/DLT/iBDU）
- ✅ 跨域时间对齐技术（解决车端多时钟源问题）
- ✅ FOTA 状态机深度理解（download → verify → install → reboot）
- ✅ 多智能体协同推理（Log + Jira + Doc 三路证据融合）
- ✅ 可追溯的证据链输出（每个结论都能回溯到原始日志行号）

### 1.2 业务价值

| 痛点 | 传统方式 | AI 诊断系统 |
|------|---------|------------|
| 日志规模 | 500+ 文件、5GB、数百万行，人工分析需数天 | 30 秒内完成初步分析 |
| 知识分散 | 日志、Jira、文档分散在不同系统 | 统一检索，交叉验证 |
| 经验依赖 | 依赖资深工程师经验 | 系统化沉淀历史案例 |
| 可追溯性 | 结论难以验证 | 每条结论都有引用来源 |
| 一致性 | 同问题不同工程师结论不一致 | 通过"已确认诊断缓存"保证一致性 |

## 二、技术架构深度分析

### 2.1 架构设计亮点 ⭐⭐⭐⭐⭐

#### 亮点 1：离线预处理 + 在线诊断分离

```
离线阶段（不阻塞用户）:
  日志包上传 → Parser 并发解析 → Time Alignment → Event Normalization → 入库

在线阶段（< 30 秒响应）:
  用户提问 → Router → 多 Agent 并行 → Synthesizer → 流式输出
```

**评价**: 这是正确的架构选择。避免了"用户等待日志解析"的糟糕体验。

#### 亮点 2：时间窗口裁剪快速通道（v6 新增）

```python
# 传统方案：全量解析 1GB 日志需 30 分钟
# v6 改进：故障时间已知时，仅解析 ±15 分钟窗口，2~5 分钟完成
parse_window = [fault_time - 15min, fault_time + 15min]
```

**评价**: 这是性能优化的关键突破。大部分故障排查场景都有明确的故障时间点，这个优化将首次诊断延迟从 30 分钟降到 2~5 分钟，**用户体验提升 6~15 倍**。

#### 亮点 3：编排框架选型的理性决策

文档经历了 5 个版本的迭代，最终从 LangGraph 回退到**纯 Python async**：

| 方案 | 优势 | 劣势 | 适用阶段 |
|------|------|------|---------|
| LangGraph | 复杂工作流、断点续跑、可视化 | API 不稳定、学习门槛高、调试困难 | 远期（需复杂有环图） |
| LangChain Agent | 代码简洁、LLM 自主决策 | 串行执行、流程不可控 | 中期（Agent 数量 > 5） |
| **纯 Python async** | **极稳定、完全可控、易调试** | 需手动编排 | **当前（Sprint 1~4）** ✅ |

**评价**: 这是**非常成熟的工程决策**。避免了过早引入复杂框架的风险。当前系统的编排需求（固定 DAG、无环、无多轮交互）用 `asyncio.gather` 完全够用。

#### 亮点 4：分层长期记忆设计（v6 新增）

```
第 1 层：Jira 知识库（人工审核过的历史缺陷，最高可信度）
第 2 层：已确认诊断缓存（工程师确认后的诊断结论，确保一致性）
第 3 层：原始诊断日志（所有会话归档，用于评测和审计）
```

**评价**: 这解决了"同症状不同结论"的一致性问题。通过工程师反馈闭环，系统会越用越准。

### 2.2 技术选型评估

| 技术栈 | 选型 | 评分 | 评价 |
|--------|------|------|------|
| 后端框架 | FastAPI | ⭐⭐⭐⭐⭐ | 原生 async、性能优秀、生态成熟 |
| 任务队列 | Arq | ⭐⭐⭐⭐ | 原生 async/await，比 Celery 更适合 FastAPI |

| LLM 供应商 | Claude + OpenAI | ⭐⭐⭐⭐⭐ | 双供应商 Fallback，Claude 长上下文优势明显 |
| 前端 | Next.js + React | ⭐⭐⭐⭐⭐ | 主流技术栈，SSE 流式支持好 |

**整体评价**: 技术选型**务实、稳健**，没有追求新潮技术，优先选择成熟方案。

## 三、核心技术难点与解决方案

### 3.1 难点 1：多域日志时间对齐

**问题**: 车端日志来自不同 ECU，时钟源各自独立：
- Android: wall clock（绝对时间）
- MCU: uptime（相对时间，从启动开始计数）
- DLT: 可能时间异常（1970-01-01）
- iBDU: 独立时钟

**解决方案**:
```python
# 1. 锚点事件识别：找到多个日志源中的同一物理事件
# 2. Offset 拟合：计算每个日志源的时钟偏移量
# 3. normalized_ts 生成：统一转换为绝对时间戳

# 三级降级策略：
# - 全部对齐成功 → 正常分析
# - 部分对齐失败 → 继续分析，涉及该域的结论加警告
# - 全部对齐失败 → 降级使用原始时间戳，报告顶部加醒目警告
```

**评价**: 这是**车载诊断系统的核心技术壁垒**。三级降级策略确保了系统在最坏情况下仍能输出参考结论，而不是完全阻断。

### 3.2 难点 2：LLM 429 限流防御

**问题**: 批量故障时多用户并发诊断，LLM API 速率限制成为瓶颈。

**解决方案**: 五层防御架构

```
第 1 层：语义缓存（预估 50~70% 命中率，从根源减少调用）
第 2 层：请求队列 + 优先级调度（CRITICAL > HIGH > NORMAL > LOW）
第 3 层：Key Pool 轮转（多 Key 负载均衡 + 指数退避）
第 4 层：跨供应商 Fallback（Claude ↔ OpenAI 自动切换）
第 5 层：熔断保护（触发后优雅降级，避免雪崩）
```

**评价**: 这是**生产级系统的必备设计**。特别是语义缓存层，利用了 FOTA 诊断场景"同症状高频重复"的特点，预估能减少 50~70% 的 LLM 调用。

### 3.3 难点 3：防幻觉护栏

**问题**: LLM 可能虚构引用、编造结论。

**解决方案**:
```python
# 1. 空结果兜底：Prompt 强制要求"无证据时返回 untrackable"
# 2. 引用 ID 断言验证：纯代码校验所有引用 ID 是否真实存在
# 3. 置信度量化计算：不依赖 LLM 自评，用代码计算

confidence = (
    0.4 * citation_valid_rate +      # 引用有效率
    0.2 * jira_top_similarity +      # Jira 相似度
    0.2 * log_evidence_count_score + # 日志证据数量
    0.2 * time_align_confidence      # 时间对齐可信度
)
```

**评价**: 这是**车规级可靠性要求的体现**。通过纯代码逻辑计算置信度，避免了 LLM 自评的不可靠性。

## 四、LLM API 中转方案分析

### 4.1 问题背景

Claude 和 OpenAI API 服务器在海外，中国大陆无法直接访问。

### 4.2 解决方案：LiteLLM 智能网关

**核心设计决策**: 使用 LiteLLM Proxy 作为中转层，而非简单的 Nginx 转发。

**关键优势**:

| 考量 | 哑管道（Nginx） | LiteLLM 智能网关 ✅ |
|------|----------------|-------------------|
| 跨境 Fallback | FOTA 平台处理，需多次跨境请求 | **LiteLLM 在海外本地 Fallback，一次跨境搞定** |
| Key Pool 轮转 | 需跨境传递 Key 选择信息 | **LiteLLM 在海外本地轮转** |
| 429 重试 | 每次重试都跨境 | **LiteLLM 在海外本地重试** |
| API 格式转换 | 需分别适配 Claude/OpenAI | **LiteLLM 统一转为 OpenAI 格式** |

**评价**: 这是**深思熟虑的架构选择**。把"重试/Fallback/Key 轮转"放在海外端，最大限度减少跨境请求次数。跨境网络是最脆弱的一环，这个设计显著提升了稳定性。

### 4.3 部署方案

```bash
# 美国 CN2 GIA 服务器（现有资源）
# pip + systemd 原生部署（无需 Docker/K8s）
# Nginx + Cloudflare Origin Certificate（15 年免续签）

# 链路延迟：
# 中国大陆 → 美国 CN2 GIA: ~130-180ms（跨境唯一一跳）
# 美国 VPS → Claude/OpenAI: < 10ms（同区域直连）
```

**评价**: 部署方案**简洁实用**，利用现有服务器资源，避免了额外的云服务成本。Cloudflare Origin Certificate 15 年有效期是亮点，省去了 Let's Encrypt 的定期续签麻烦。
