# Velab FOTA 智能诊断平台 - 生产环境综合部署指南

本指南阐述了如何在一台全新的 Linux 服务器（推荐 Ubuntu 20.04/22.04 LTS）上，以“一键化”和“纯白匣”的方式，安全高可用地部署 FOTA 诊断平台的全部三大子模块（**前端 Web**、**后端 Backend**、**大模型代理 Gateway**）。

---

## 1. 架构总览与自动包容

本部署体系秉承**全自动按需装载**设计：
您无需提前手工费力配置数据库和开发语言，系统集成的自动化脚本会在第一阶段自动调用 `apt` 拉取并安装所有缺失的核心运行库层：
* **核心运行组件**：`Python 3 (包含 venv & pip)` 及其对应的 `Node.js (含 npm)`
* **数据库/缓存中间件**：`PostgreSQL`, `Redis-Server`

应用会最终分别沉降到宿主机的专属隔离目录下，以专属无权限子账号静默运行：
* 网关模块 (`litellm-proxy`) -> 运行于 `:4000` 端口
* 后端模块 (`fota-backend`) -> 运行于 `:8000` 端口
* 前端模块 (`fota-web`) -> 运行于 `:3000` 端口

---

## 2. 极速一键部署 (推荐)

如果您希望在单独的一台服务器（或开发/测试容器环境）上进行全家桶快速挂载，我们提供了一键通用的全局联合脚本：

```bash
# 1. 赋予执行权限（如果全新 pull 的代码无权限）
chmod +x scripts/deploy-all.sh

# 2. 运行一键配置（必须使用 sudo 超级管理员）
sudo ./scripts/deploy-all.sh
```

**交互选项：**
在安装过程中，您需要指定部署形态矩阵：
* **场景 A**：平台部署在国内，通过本机启动网关组件解决网络不畅或分发中转的问题。
* **场景 B**：直连模式（平台部署在海外）。后端直接通讯于外部云提供商，前端与后端只安装 2 个组件（跳过网关）。

等待流程（包括下载构建、生成 Schema 表结构等）执行到 100% 结束。

---

## 3. 按组件手动分发部署 (进阶/分布式)

对于微服务或者内外网分离多层集群的情况，我们为每个子工程独立了专属的构建流，您可以把不同项目下发到不同的独立主机上：

### 3.1 部署大语言模型网关 (Gateway)
> 内网映射与负载均衡的第一屏障，采用 LiteLLM。
```bash
cd gateway
sudo bash scripts/deploy.sh
```
部署地：`/opt/litellm-proxy/`
专属用户：`litellm`
该进程会在隔离安全沙盒里运作，具备本地挂载 SQLite 生成 Token Metrics 的自理权限。

### 3.2 部署诊断平台后端 (Backend)
> 驱动 Agent 推理与业务底座，包含数据库连接。
```bash
cd backend
sudo bash scripts/deploy.sh
```
部署地：`/opt/fota-backend/`
专属用户：`fota`
*注：此脚本将自动触发内置的 `init_postgres.sh`，处理 PostgreSQL 系统级别用户的生成与数据库 `fota_db` 及数据表结构的填充灌入，全程免人工干预。*

### 3.3 部署 Web 前端展示层 (Frontend)
> 基于 Next.js 生产态封装流式传输响应层。
```bash
cd web
sudo bash scripts/deploy.sh
```
部署地：`/opt/fota-web/`
专属用户：`fota-web`
脚本会屏蔽 `node_modules` 回滚并自动接盘依赖清洗，触发服务端编译 (SSR Buid)。

## 4. 后续配置与生命周期管理

### 4.1 配置文件修正
由于部署是自动化且带有安全机制的，系统对密码和涉及金融余额的 API 秘钥做了留白处理。

**💡 自动同步功能说明：**
如果您使用 `deploy-all.sh` 部署，系统会自动为您生成并同步 `LITELLM_MASTER_KEY` 与 `LITELLM_API_KEY`（内部鉴权密钥），您**无需手动对准**。

你需要修改生成在目标服务器位置下的配置集：

**1. Gateway 网关路由验证 (场景 A 下尤为重要)：**
编辑 `/opt/litellm-proxy/.env`
填入需要代转交的真实模型供应商 API 秘钥。

**2. Backend 数据库与模型挂接：**
编辑 `/opt/fota-backend/.env`
修正/核对数据库默认无密码直连的可行性；并将内部大模型通讯地址指向已运转的 API 网址或本机 Gateway 地址（http://127.0.0.1:4000）。

**3. Web 端跨域配置：**
编辑 `/opt/fota-web/.env.local`
将 `NEXT_PUBLIC_BACKEND_URL` 设置为外部用户可直接触达的后端公网 IP 或域名，而非内网地址。

*(注：系统底层安全策略拒绝跨域组件篡改配置，每次修改过这些文件，请务必执行重启指令)*

### 4.2 通过 systemd 进行运维管控
部署结束后，应用已全面接管至 `systemctl`，您可以以管理员身份无缝执行状态排查与守护：

**启动应用**
```bash
sudo systemctl start litellm
sudo systemctl start fota-backend
sudo systemctl start fota-web
```

**应用下线止血**
```bash
sudo systemctl stop {service-name}
```

**健康心跳监测**
```bash
sudo systemctl status {service-name}
```

**翻阅报错与进程日志**
系统抛弃了老旧的文件写覆盖模式，全量日志归档通过原生 System Journal 呈现：
```bash
# 滚动实时查看（类似 tail -f）
sudo journalctl -u fota-backend -f

# 查看近半小时报错记录
sudo journalctl -u fota-web --since "30 min ago"
```

---

*这份指南随脚本自动化演进保持实时状态同步，如有运行异常问题，请确认端口（3000、8000、4000、5432）是否遭遇外部云盾拦截或被本地其他 Docker 残留占据。*
