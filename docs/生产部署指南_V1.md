# Velab FOTA 智能诊断平台 - 生产环境综合部署指南

本指南阐述了如何在一台全新的 Linux 服务器（推荐 Ubuntu 20.04/22.04 LTS）上，以“一键化”和“纯白匣”的方式，安全高可用地部署 FOTA 诊断平台的全部三大子模块（**前端 Web**、**后端 Backend**、**大模型代理 Gateway**）。

---

## 1. 架构总览与自动包容

本部署体系秉承**全自动按需装载**设计：
您无需提前手工费力配置数据库和开发语言，系统集成的自动化脚本会在第一阶段自动调用 `apt` 拉取并安装所有缺失的核心运行库层：
* **核心运行组件**：`Python 3 (包含 venv & pip)` 及其对应的 `Node.js (含 npm)`
* **数据库/缓存中间件**：`PostgreSQL`, `Redis-Server`

应用会最终分别沉降到宿主机的专属隔离目录下，以专属无权限子账号静默运行：
* 网关模块 (`litellm-proxy`) -> 运行于 `:4000` 端口
* 后端模块 (`fota-backend`) -> 运行于 `:8000` 端口
* 前端模块 (`fota-web`) -> 运行于 `:3000` 端口

---

## 2. 极速一键部署 (推荐)

如果您希望在单独的一台服务器（或开发/测试容器环境）上进行全家桶快速挂载，我们提供了一键通用的全局联合脚本：

```bash
# 1. 赋予执行权限
chmod +x scripts/deploy-all.sh

# 2. 交互式运行（按提示选择场景和域名）
sudo ./scripts/deploy-all.sh

# 或：非交互式运行（适合 CI/CD 或自动化场景）
sudo ./scripts/deploy-all.sh --mode A --domain your-server.com
sudo ./scripts/deploy-all.sh --mode B --domain your-server.com
```

**参数说明：**
| 参数 | 说明 | 示例 |
|------|------|---------|
| `--mode A` | 场景 A：国内部署，通过本机 Gateway 中转 LLM | `--mode A` |
| `--mode B` | 场景 B：海外部署，Backend 直连 LLM API | `--mode B` |
| `--domain` | 平台访问地址（域名或公网 IP），留空默认 `localhost` | `--domain example.com` |

等待流程（包括下载构建、生成 Schema 表结构等）执行到 100% 结束。

## 3. 按组件手动分发部署 (进阶/分布式)

对于微服务或者内外网分离多层集群的情况，我们为每个子工程独立了专属的构建流，您可以把不同项目下发到不同的独立主机上：

### 3.1 部署大语言模型网关 (Gateway)
> 内网映射与负载均衡的第一屏障，采用 LiteLLM。
```bash
cd gateway
sudo bash scripts/deploy.sh
```
部署地：`/opt/litellm-proxy/`
专属用户：`litellm`
该进程会在隔离安全沙盒里运作，具备本地挂载 SQLite 生成 Token Metrics 的自理权限。

### 3.2 部署诊断平台后端 (Backend)
> 驱动 Agent 推理与业务底座，包含数据库连接。
```bash
cd backend
sudo bash scripts/deploy.sh
```
部署地：`/opt/fota-backend/`
专属用户：`fota`
*注：此脚本将自动触发内置的 `init_postgres.sh`，处理 PostgreSQL 系统级别用户的生成与数据库 `fota_db` 及数据表结构的填充灌入，全程免人工干预。*

### 3.3 部署 Web 前端展示层 (Frontend)
> 基于 Next.js 生产态封装流式传输响应层。
```bash
cd web
sudo bash scripts/deploy.sh
```
部署地：`/opt/fota-web/`
专属用户：`fota-web`
脚本会屏蔽 `node_modules` 回滚并自动接盘依赖清洗，触发服务端编译 (SSR Buid)。

## 4. 后续配置与生命周期管理

### 4.1 配置文件修正

**🔐 自动生成的密钥（无需手动填写）：**

| 变量 | 自动生成时机 | 说明 |
|------|------------|------|
| `POSTGRES_PASSWORD` | `backend/scripts/deploy.sh` Step 6 | 检测到弱密码 `fota_password` 时自动替换为 40 字符强随机值，并通过 `init_postgres.sh` 同步写入数据库 |
| `LITELLM_MASTER_KEY` | `deploy-all.sh` Step 5 | 检测到占位值时自动生成并双向同步到 `gateway/.env` 和 `backend/.env` 的 `LITELLM_API_KEY` |
| `BACKEND_URL` / `NEXT_PUBLIC_BACKEND_URL` | `deploy-all.sh` Step 5 | 根据 `--domain` 参数自动写入 |

**✏️ 必须手动填写的变量（脚本无法代劳）：**

**1. Gateway LLM API Keys（场景 A 必填）：**
```bash
sudo nano /opt/litellm-proxy/.env
```
填入以下变量的真实值（部署脚本会在结束时列出哪些仍为占位值）：
- `ANTHROPIC_API_KEY` — router-model (Haiku) 专用
- `ANTHROPIC_API_KEY_1` — agent-model 和 synthesizer-model 共用
- `ANTHROPIC_API_KEY_2` — agent-model 轮转池第二个 Key（防御 429）
- `OPENAI_API_KEY` — 所有模型的 Fallback

**2. Backend LLM API Keys（场景 B 必填）：**
```bash
sudo nano /opt/fota-backend/.env
```
填入 `ANTHROPIC_API_KEY` 或 `OPENAI_API_KEY` 至少一个。

**3. Web 端（通常由 `deploy-all.sh` 自动处理，仅单独部署时需手动）：**
```bash
sudo nano /opt/fota-web/.env.local
```
将 `NEXT_PUBLIC_BACKEND_URL` 设置为浏览器可访问的后端地址。

> 每次修改配置文件后需重启对应服务：`sudo systemctl restart <service-name>`

### 4.2 通过 systemd 进行运维管控
部署结束后，应用已全面接管至 `systemctl`，您可以以管理员身份无缝执行状态排查与守护：

**启动应用**
```bash
sudo systemctl start litellm
sudo systemctl start fota-backend
sudo systemctl start fota-web
```

**应用下线止血**
```bash
sudo systemctl stop {service-name}
```

**健康心跳监测**
```bash
sudo systemctl status {service-name}
```

**翻阅报错与进程日志**
系统抛弃了老旧的文件写覆盖模式，全量日志归档通过原生 System Journal 呈现：
```bash
# 滚动实时查看（类似 tail -f）
sudo journalctl -u fota-backend -f

# 查看近半小时报错记录
sudo journalctl -u fota-web --since "30 min ago"
```
